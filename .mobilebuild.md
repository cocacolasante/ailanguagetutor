## Stack decision — use this, do not re-evaluate

**React Native with Expo (SDK 51+) + TypeScript**

Reasons:
- Fastest path to App Store + Play Store from a single codebase
- EAS Build handles iOS builds without requiring macOS
- `expo-av` covers TTS playback and microphone input
- `expo-secure-store` handles JWT storage securely on both platforms
- `react-native-sse` handles the SSE streaming conversation API
- Stripe mobile billing via `@stripe/stripe-react-native`
- Expo Router (file-based routing) for clean navigation
- Excellent TypeScript support throughout

**Do not use bare React Native workflow. Do not use Flutter.**

## Billing strategy — critical

**Do NOT implement Apple IAP or Google Play Billing for the initial mobile app.**

The existing Stripe subscription is web-originated and managed by the Go backend. The correct mobile approach is:

1. Check `/api/billing/status` on app load — show paywall if no active subscription
2. On upgrade tap → POST `/api/billing/checkout` → get Stripe URL → open in system browser via `expo-linking`
3. On return to app → check `/api/billing/verify-checkout?session_id=` or just re-fetch `/api/billing/status`
4. The Go backend webhooks handle all subscription state changes

This avoids Apple/Google's 30% cut and is valid for web-based SaaS apps. The App Store allows this as long as you do not offer a native in-app purchase alternative.

## SSE streaming — critical implementation note

The conversation screen must stream AI responses token by token via SSE. Implement like this:

```typescript
// src/hooks/useConversationStream.ts
import EventSource from 'react-native-sse';

export function useConversationStream() {
  const stream = (sessionId: string, message: string, token: string, onToken: (t: string) => void) => {
    const es = new EventSource(`${API_BASE}/api/conversation/message`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${token}`,
      },
      body: JSON.stringify({ sessionId, message }),
    });
    es.addEventListener('message', (e) => onToken(e.data ?? ''));
    es.addEventListener('done', () => es.close());
    es.addEventListener('error', () => es.close());
    return () => es.close();
  };
  return { stream };
}
```

## Voice/audio approach

- **TTS playback:** POST to `/api/tts` → returns audio stream → play with `expo-av` `Audio.Sound.createAsync()`
- **Microphone input:** `expo-av` `Audio.Recording` → transcribe via `expo-speech` or send audio blob to future `/api/stt` endpoint
- **Permissions:** Request microphone permission at conversation start using `Audio.requestPermissionsAsync()`
- **Background audio:** Set `Audio.setAudioModeAsync({ playsInSilentModeIOS: true })` so TTS works when phone is silenced
- **Phase 1:** TTS playback only (tap-to-hear button on AI messages)
- **Phase 2:** Voice input via microphone (record → send text transcript)
- **Phase 3:** Real-time pronunciation feedback (requires new backend endpoint)

## Risks and known gaps

| Issue | Mitigation |
|---|---|
| No `/api/stt` endpoint for voice input | Phase 1: text input only. Phase 2: use expo-speech device STT or add backend endpoint |
| Email verification requires browser | After register, show a screen instructing user to check email; deep link back with `expo-linking` when they tap verify link |
| Stripe checkout requires browser | Open with `expo-linking.openURL()` — this is intentional and correct |
| SSE in React Native | Use `react-native-sse` — do not use browser EventSource |
| ElevenLabs audio is a stream | Pipe to a temp file or use `Audio.Sound.createAsync()` with URI — test buffer size |
| iOS App Store review | Ensure billing is clearly a web subscription; include "Manage Subscription" link to Stripe portal |
