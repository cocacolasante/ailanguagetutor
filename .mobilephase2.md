## Backend context — practice modes and gamification

**Important:** The backend uses **Redis** for in-memory session state across 
all practice modes. Each practice session is created server-side and tracked 
by a session ID. The mobile app must:
- Always start a session before sending answers
- Always complete a session (even on back-navigation) to avoid orphaned Redis sessions
- Handle session expiry gracefully — if the backend returns a 404 or session-expired 
  error mid-practice, show a friendly "Session expired, start again" state and 
  navigate back to the practice selection screen
- please update any section from phase 1 that should use redis, use the web app backend to reference what needs redis

## API reference — practice modes

### Vocab
```
POST /api/vocab/session
  body: { language, level }
  returns: { sessionId, words: [{ word, translation, example }] }

POST /api/vocab/check
  body: { sessionId, word, answer }
  returns: { correct: bool, correctAnswer: string, explanation: string }

POST /api/vocab/word-result
  body: { sessionId, word, correct: bool }
  returns: { ok: bool }

POST /api/vocab/complete
  body: { sessionId }
  returns: { score, total, fluencyPoints, weakVocab: string[] }
```

### Sentences
```
POST /api/sentences/session
  body: { language, level }
  returns: { sessionId, sentences: [{ prompt, hint }] }

POST /api/sentences/check
  body: { sessionId, sentenceIndex, answer }
  returns: { correct: bool, correctAnswer: string, feedback: string }

POST /api/sentences/complete
  body: { sessionId }
  returns: { score, total, fluencyPoints }
```

### Listening
```
POST /api/listening/session
  body: { language, level }
  returns: { sessionId, exercises: [{ audioText, question, options: string[] }] }
  Note: audioText is what should be sent to /api/tts for playback

POST /api/listening/complete
  body: { sessionId, answers: [{ exerciseIndex, answer }] }
  returns: { score, total, fluencyPoints, correctAnswers: string[] }
```

### Writing
```
POST /api/writing/session
  body: { language, level }
  returns: { sessionId, prompt: string, targetLanguage: string }

POST /api/writing/message
  body: { sessionId, message }
  returns: SSE stream (same pattern as /api/conversation/message)
  Use react-native-sse — same implementation as conversation streaming

POST /api/writing/complete
  body: { sessionId }
  returns: { feedback: string, fluencyPoints, corrections: string[] }
```

### Gamification / improvement
```
GET /api/user/stats
  returns: {
    streak: number,
    totalFP: number,
    languageFP: { it: number, es: number, pt: number },
    achievements: [{ id, name, description, earnedAt }],
    recentConversations: [...],
    weakAreas: string[],
    strongAreas: string[],
    recentTopics: string[],
    recentVocab: string[],
    nextSuggestions: string[],
    weakVocab: string[],
    weakGrammar: string[]
  }

GET /api/user/mistakes
  returns: { mistakes: [{ type, description, example, count }] }

GET /api/conversation/records
  returns: [{ id, language, level, topic, messageCount, fluencyPoints, summary, createdAt }]

GET /api/conversation/records/:id
  returns single record with full message history

GET /api/badges
  returns: [{ id, name, description, icon, earnedAt | null }]

GET /api/leaderboard
  returns: [{ username, totalFP, level, rank }] — top 50, public endpoint (no auth needed)
```

### 10. Vocab Practice — `app/(app)/practice/vocab.tsx`

**Flow:**
1. Start session → POST `/api/vocab/session` with selected language + level
2. Show words one at a time as flashcard-style cards
3. Each card: show the word in target language → user types translation → submit
4. POST `/api/vocab/check` → show correct/incorrect with explanation
5. POST `/api/vocab/word-result` to record result
6. After all words → POST `/api/vocab/complete` → show results screen inline

**UI:**
- Large word display card (target language word, big font)
- Text input for translation answer
- "Check" button
- Animated flip or slide to reveal correct answer
- Progress bar showing word N of total
- Score tracker (X correct / Y total) in header

**Session cleanup:**
- On back navigation before complete → POST `/api/vocab/complete` anyway 
  (partial completion is better than orphaned Redis session)
- On session expiry (404 from backend) → show "Session expired" inline error, 
  offer "Start New Session" button

**Results (inline, same screen after complete):**
- Score: X/Y correct
- FP earned
- Weak vocab list (words to review)
- "Practice Again" and "Back to Practice" buttons

### 12. Listening Comprehension — `app/(app)/practice/listening.tsx`

**Flow:**
1. POST `/api/listening/session` → get exercises with audioText + question + options[]
2. For each exercise:
   a. Show "Play Audio" button → POST `/api/tts` with audioText → play with expo-av
   b. Show multiple choice options
   c. User taps answer
3. Collect all answers
4. POST `/api/listening/complete` with all answers → show results

**UI:**
- Large "▶ Play Audio" button (prominent, center of screen)
- "Replay" button after first play
- Loading indicator while TTS is fetching/buffering
- Multiple choice options as tappable cards (highlight selected)
- "Submit Answer" button
- Progress indicator: Exercise N of total

**Audio handling (important):**
```typescript
// Pattern to follow in useAudio hook or inline:
import { Audio } from 'expo-av';

await Audio.setAudioModeAsync({ playsInSilentModeIOS: true });

const response = await apiClient.post('/api/tts', 
  { text: exercise.audioText, language },
  { responseType: 'arraybuffer' }
);
// Write to FileSystem cache then play
import * as FileSystem from 'expo-file-system';
const uri = FileSystem.cacheDirectory + 'tts_audio.mp3';
await FileSystem.writeAsStringAsync(uri, 
  Buffer.from(response.data).toString('base64'), 
  { encoding: FileSystem.EncodingType.Base64 }
);
const { sound } = await Audio.Sound.createAsync({ uri });
await sound.playAsync();
```

Add `expo-file-system` to dependencies: `npx expo install expo-file-system`

**Results (inline):**
- Score, FP, which answers were correct vs wrong
- Show correct answers for any missed questions

## Redis session lifecycle — handle these cases

The Go backend stores all practice sessions in Redis with a TTL. Every practice 
screen must handle these error states:

| HTTP status | Meaning | Mobile behavior |
|---|---|---|
| 404 on `/check` or `/complete` | Session expired in Redis | Show "Session expired" error state, offer restart |
| 400 | Bad request (wrong answer format etc.) | Show inline validation error |
| 401 | JWT expired | Auth interceptor handles redirect to login |
| 500 | Backend error | Show generic error with retry button |

Create a shared error handler util if not already in `src/utils/errors.ts`:

```typescript
export function isPracticeSessionExpired(error: unknown): boolean {
  return axios.isAxiosError(error) && error.response?.status === 404;
}
```

## State management for practice sessions

Use local `useState` within each practice screen — do NOT put practice session 
state in Zustand. Practice sessions are ephemeral and screen-local. Pattern:

```typescript
const [sessionId, setSessionId] = useState<string | null>(null);
const [currentIndex, setCurrentIndex] = useState(0);
const [answers, setAnswers] = useState<Answer[]>([]);
const [phase, setPhase] = useState<'loading' | 'practice' | 'results' | 'error'>('loading');
```

The only global state that changes after a practice session completes is 
`userStats` — invalidate that TanStack Query key after any session `.complete()` 
call so the dashboard and improve screen reflect updated FP and weak areas:

```typescript
queryClient.invalidateQueries({ queryKey: queryKeys.userStats });
```
